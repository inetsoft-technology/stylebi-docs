If you are using the *default security provider* for authentication, you can define users in Enterprise Manager as described below. If you are using an LDAP server for authentication, you should define users in your LDAP software. See xref:SpecifySecurityProvider.adoc[] for more information about security providers.


[TIP]
====
[square]
* There are two embedded users: *admin* is a super-user with the role of ‚ÄòAdministrator‚Äô and *guest* is a user with the role of ‚ÄòEveryone‚Äô. The default passwords are ‚Äòadmin‚Äô and ‚Äòsuccess123‚Äô, respectively.
* To allow users to log in without a password (as  ‚Äúguest‚Äù), create a user with name *anonymous*.  This will allow all users to log in without entering credentials and be governed by the permissions assigned to the ‚Äòanonymous‚Äô user (see xref:SetRepositoryPermissions.adoc[] and xref:SetSecurityActions.adoc[]). If  a user wants to log in under their own name, they will need to press the {buttonAccount} and select {buttonLogin}.
====



== Create a New User
To create a new user in Enterprise Manager , follow the steps below:



. Press the {buttonSettings} at the top of Enterprise Manager.
. Select the *Security* page in the left panel.
+
image::Figure 23_1.png[alt="select Security page in left panel"]
. Select the *Users* tab.
+
image::securityDefineUsers.png[alt="select Users tab in Security page"]

. From the ‚ÄòProvider Name‚Äô menu, select the security provider under which to define users.

include::administration:partial$selectOrganization.adoc[]
+
New users will be added to the organization you select.

. Press the ‚ÄòNew User‚Äô link below the tree. This creates a user with the default name ‚ÄòuserN‚Äô (user0, user1, etc.).
+
image::server1.png[alt="new user created in security tree"]

. Type in the ‚ÄòName‚Äô field to change the name of the user.


. *_Optional:_* Enter an ‚ÄòAlias‚Äô for the user. This is the name that will be displayed in the Portal for the user.

. Select ‚ÄòActive‚Äô from the right-side menu to allow the user to log into the Portal.
+
NOTE: When a user is *not* marked as '`Active`', the user  cannot log in, but all assets and scheduled tasks continue to function  normally.

. Specify an email address for the user in the ‚ÄòEmail‚Äô field.

. *_Optional_*:  Select a locale for the user from the '`Locale`' menu.  See xref:RegisterLocale.adoc[]  for information on how to define  the locales that appear in this menu.  See xref:Localization.adoc[] for full information on localization.

. To select a Portal theme for the user, choose the desired theme from the ‚ÄòTheme‚Äô menu. (See xref:Presentation.adoc[] for information about how to create a theme.)
include::ROOT:partial$enterpriseOnlyInline.adoc[]
+
IMPORTANT: The multi-tenant feature in Enterprise Edition must be *disabled*.  See xref:EnableMultiTenant.adoc[] for more information.

. Select the ‚ÄòChange Password‚Äô option to change the user‚Äôs password.
. *_Optional_*: To add the user to a group, press the btn:[Add] button in the ‚ÄòMember Of‚Äô area. Choose the desired group(s) in the ‚ÄòAdd Group‚Äô dialog box (kbd:[Ctrl]-click to select multiple groups), and press btn:[Add].
+
image::securityDefineUsers2.png[alt="add user to group in Member Of area"]
. *_Optional:_* To add a role for the user, press the btn:[Add] button in the ‚ÄòRoles‚Äô area. Choose the desired role(s) in the ‚ÄòAdd Role‚Äô dialog box (kbd:[Ctrl]-click to select multiple roles), and press btn:[Add].

. *_Optional:_* To add administrative permission for this user (ability for other users to administer this user), press the btn:[Add] button in the ‚ÄòAdministrator Permissions‚Äô area. Choose the desired user(s), groups(s), and role(s)  in the ‚ÄòAdd Permission‚Äô dialog box (kbd:[Ctrl]-click to select multiple items), and press btn:[Add]. Users, groups, or roles granted such administrative privileges will be able to use Enterprise Manager to remove the defined user or reassign groups and roles to the user.



. Press btn:[Apply] to save your changes.

. Repeat the above steps to add additional users. If you are using a ‚ÄúNamed Viewer Session‚Äù license, the number of users you can add is limited by the license.  See note below.
+
[IMPORTANT]
.Defining Users for Named Licenses
====
If you are using  a ‚ÄúNamed Viewer Session‚Äù or ‚ÄúNamed Session‚Äù license key, the security provider must define the named users. (See xref:SpecifySecurityProvider.adoc[] for more information about security provider options.)

* If you are using the default InetSoft security provider, you will not be permitted to create more users than the installed licenses allow. If existing users already exceed the licensed number, remove the excess users and restart the server.

* If you are using an LDAP security provider, add a user filter to return only the desired named users. For example, `(|(uid=user1)(uid=user2)(uid=user3))` returns only user1, user2, and user3.
* If you are using a custom security provider, modify the implementing class to ensure that the number of users is within the licensed limit.

Note that if any security provider defines more than the allowed number of users, it is not possible to predict which of those users will be granted access to the User Portal.

For a ‚ÄúNamed Session‚Äù license key (which enables Visual Composer), you should specify which particular users have permission to access Visual Composer. To do this, assign the ‚ÄòDesigner‚Äô role to users who need Visual Composer access.

image::NamedSessionDesignerRole.png[alt="assign Designer role to user for Visual Composer access"]

You can assign the ‚ÄòDesigner‚Äô role to more users than the maximum number allowed by the license. If the maximum number of licensed users is reached, the next named user who attempts to open Visual Composer will receive a notification that all licenses are in use.
====



== Edit an Existing User

To edit a user, follow the steps below:

include::administration:partial$selectOrganization.adoc[]

. Expand the user`'s organization on the tree,  expand the ‚ÄòUsers‚Äô node, and click on the username to select it. (A user may be listed directly under the ‚ÄòUsers‚Äô node or be listed within a group under the ‚ÄòUsers‚Äô node.)

. Make any desired changes as described above, and press btn:[Apply].

NOTE: For efficiency, the number of users listed in the navigation tree under LDAP security is limited by the `em.user.max` property (default 1000). See xref:AllProperties.adoc[] for more information about setting properties. Unlisted users can be found by using the {buttonFilterField}.


[#CopyandPasteIdentities]
== Copy and Paste Identities

IMPORTANT: You cannot copy and paste identities across different organizations or different security providers.

To copy the '`Member Of`', '`Roles`', or '`Administrator Permissions`' identities from one user to another, follow the steps below:

. Press the {buttonSettings} at the top of Enterprise Manager.
. Select the *Security* page in the left panel.
+
image::Figure 23_1.png[alt="select Security page in left panel"]
. Select the *Users* tab.
+
image::securityDefineUsers.png[alt="select Users tab in Security page"]

. From the ‚ÄòProvider Name‚Äô menu, select the security provider under which to define users.

include::administration:partial$selectOrganization.adoc[]

. Press the btn:[Copy Identities] button below the '`Member Of`', '`Roles`', or '`Administrator Permissions`' tables to copy the current identities. You can then navigate to another user and paste those identities.

. Press the btn:[Paste Identities] button below the corresponding table to paste the identities you have copied.
+
TIP: The btn:[Paste Identities] button will display numbers in parentheses, such as btn:[Paste Identities (x of n)]. The *second value* (*n*) indicates the number of identities that were copied to the clipboard. The *first value* (*x*) indicates the number of identities that are compatible with the current user, and can actually be pasted.

. Press btn:[Apply] to save your changes.
